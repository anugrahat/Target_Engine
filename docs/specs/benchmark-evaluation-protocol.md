# Benchmark Evaluation Protocol

## Purpose

This document defines how PrioriTx benchmark cases must be evaluated before any
model or policy layer is treated as scientifically meaningful.

The goal is to outperform the public PandaOmics baseline on reproducibility and
decision quality without introducing hidden leakage or inflated success claims.

## Scope

This protocol applies to:

- benchmark packs in `configs/indications/`
- dataset manifests and study contrasts in `data_contracts/`
- baseline rankers
- graph-augmented rankers
- any later learning-to-rank or bandit layer

## Core Principles

- Evaluate discovery-time evidence only.
- Keep validation strength explicit instead of collapsing everything into a single success label.
- Separate target nomination performance from molecule-generation or chemistry success.
- Distinguish translational evidence from mechanistic evidence.
- Record unresolved uncertainty instead of imputing confidence.

## Benchmark Units

Each benchmark case must declare:

- `benchmark_id`
- indication definition
- accession-backed discovery datasets
- allowed validation evidence
- forbidden evidence sources
- known-positive targets if and only if they are source-backed

The current implementation materializes source-backed positives as JSON
assertion files under `data_contracts/assertions/`.

## Label Tiers

PrioriTx should not use a single binary target-success label across all cases.

Use tiered labels instead:

### Tier 1: Mechanistic Support

Examples:

- pathway reversal in relevant tissue or cells
- CRISPR or perturbation evidence
- rescue in disease-relevant model systems

### Tier 2: Translational Support

Examples:

- reproducible signal across independent cohorts
- orthogonal omics support
- human genetics support

### Tier 3: Programmatic Advancement

Examples:

- compound nomination
- in vivo efficacy
- IND-enabling progression
- clinical-trial entry

Tier 3 evidence is useful for narrative context, but it must not be used as a
discovery-time training label by default.

## Forbidden Leakage

Do not use the following as discovery-time features when they occur downstream
of target nomination:

- clinical-trial entry for the nominated target
- medicinal chemistry success
- AlphaFold-enabled tractability work done after target selection
- later experimental validation generated by the same paper being benchmarked
- prose from the discussion section describing why the chosen target succeeded

These may be evaluation annotations, not ranking inputs.

## Primary Metrics

For target ranking:

- NDCG@10
- NDCG@25
- MAP
- Precision@5
- Precision@10
- Recall@25

For enrichment and calibration:

- enrichment over source-backed positives
- score calibration curves for top-ranked targets
- ablation deltas by evidence family

## Benchmark Reporting Table

Every benchmark run must report:

- benchmark pack version
- dataset manifest versions used
- target universe size
- number of source-backed positives
- label tier used for scoring
- metrics by tier
- top 25 targets with provenance

## Case-Specific Rules

### IPF / TNIK

- Discovery evaluation must be based on the disclosed transcriptomic cohorts.
- Clinical-trial progression is a downstream annotation, not a discovery-time label.
- Blood and lung contrasts should be reported separately unless a heterogeneity model is declared.

### HCC / CDK20

- Discovery evaluation must exclude AlphaFold and Chemistry42 downstream success from the feature set.
- Target ranking should be evaluated separately from chemistry-generation outcomes.
- Adjacent-normal tissue should be flagged as a possible non-healthy control source.

### ALS

- `Answer ALS` inputs remain partially specified and should be treated as manifest-incomplete until stable identifiers are recovered.
- Drosophila rescue evidence belongs in mechanistic support, not direct clinical-translational support.

### AD Phase Separation

- The case remains partial until supplemental datasets are recovered.
- Do not score this benchmark as accession-complete.

## Split Strategy

Use the strongest available split that the case supports:

1. time-aware split if discovery and validation timestamps are explicit
2. study-held-out split across cohorts
3. evidence-family ablation when only one positive target is available

If none of these are possible, the benchmark must be marked `limited_evaluation`.

## Output Requirements

Every scored target must emit:

- `target_rank_record`
- contributing feature components
- supporting dataset references
- risk flags
- explicit statement of which evidence tiers were used for evaluation

## Exit Criteria

A benchmark is implementation-ready only when:

1. its benchmark pack validates
2. at least one dataset manifest validates
3. at least one study contrast validates
4. the allowed and forbidden evidence lists are written
5. the benchmark label tier is explicit

Without these, the case remains research-only.
